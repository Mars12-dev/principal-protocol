# Principal Protocol Technical Specification

## 1. Overview

Principal Protocol is a Soroban-native yield tokenization protocol for Stellar. It is designed as a general split-principal and future-yield architecture for yield-bearing assets, with permissioned RWAs such as Ondo USDY used as the first reference case. The protocol separates a yield-bearing asset position into:

- **Principal Token (PT)**: a fixed-income instrument redeemable for the principal value at maturity.
- **Yield Token (YT)**: a claim on the future yield generated by the underlying asset until maturity.

The protocol supports Stellar yield-bearing assets and Stellar Asset Contract (SAC) representations while preserving strong authorization, deterministic settlement, and explicit risk controls.

## 2. Goals

- Enable fixed-income exposure from Stellar yield-bearing assets without requiring users to hold the underlying variable-yield position.
- Enable direct yield exposure and hedging by separating principal and future-yield claims.
- Build on Stellar’s contract-account model and existing yield asset ecosystem.
- Support permissioned compliance flows for restricted assets when required.
- Launch with USDY and extend to additional yield-bearing assets.

## 3. Core Components

### 3.1 Protocol Components

- **SY Wrapper**: wraps the underlying yield-bearing asset into a standardized yield wrapper (e.g. `SY-USDY`).
- **Tokenization Engine**: mints and burns PT and YT from the wrapped asset, manages maturity rules, and settles at maturity.
- **PT/SY Market**: a DEX pool that enables trading of PT against the wrapped asset and indirect trading of YT.
- **Router / Settlement Layer**: abstracts deposit, minting, trading, and redemption flows.
- **Oracle & Reference Value Module**: provides the underlying asset reference price for pricing and maturity settlement.
- **Permissioning Module**: enforces eligibility rules inherited from permissioned underlying assets when applicable.
- **Risk & Control Module**: includes stale-price checks, circuit breakers, and emergency pause logic.

## 4. Asset Model

### 4.1 Underlying Asset

The initial supported asset is **USDY**. USDY is a permissioned, yield-bearing token on Stellar. Principal Protocol is designed to support USDY as the first reference asset, but the architecture is asset-agnostic and may be extended to other yield-bearing tokens and fixed-income instruments.

Principal Protocol assumes:

- the underlying asset is represented as a valid Stellar Asset Contract;
- supply and transfer permissions are enforced by issuer and permissioning logic when needed;
- the protocol preserves these permission constraints for all derived instruments.

### 4.2 Wrapped Asset: SY-USDY

`SY-USDY` is a standardized wrapper around USDY that:

- holds the underlying USDY position;
- tracks reference-value accounting in USDC terms;
- enables uniform issuance of PT and YT.

The wrapper must support: deposit, withdrawal, and reference-value updates.

### 4.3 Principal Token (PT)

PT represents the fixed principal claim at maturity.

- Denominated in USDC accounting value.
- Redeemable at maturity for the principal value converted into underlying USDY using the reference value at maturity.
- Can trade before maturity at a discount or premium relative to its maturity redemption value.

### 4.4 Yield Token (YT)

YT represents the future yield claim.

- Captures the excess value above the principal benchmark at maturity.
- Trades as a separate instrument representing future yield exposure.
- Settlement at maturity depends on realized underlying reference value.

## 5. Lifecycle and Flows

### 5.1 Deposit and Minting

1. User deposits USDY into the SY wrapper.
2. The wrapper standardizes the position and records the reference value at deposit or issuance.
3. The tokenization engine mints:
   - PT tokens equal to the principal value of the deposited position.
   - YT tokens equal to the future yield claim.

### 5.2 Trading

- PT is listed in a PT/SY liquidity pool.
- Users can buy or sell PT directly against the wrapped underlying asset.
- YT trading is handled through a routing layer that mints/redeems YT around PT/SY liquidity operations.

### 5.3 Redemption at Maturity

At maturity:

1. The protocol computes the reference-value of the underlying USDY in USDC.
2. PT holders redeem for the principal value expressed in underlying USDY.
3. YT holders redeem for the realized yield value in underlying USDY.
4. Both token classes are burned and underlying assets are distributed.

### 5.4 Recombination

The protocol must support recombining PT and YT prior to maturity if both instruments are held by the same participant, enabling a full underlying exposure restore.

## 7. Risk Controls and Security

### 7.1 Oracle Security

- The USDY/USDC reference price oracle must be permissioned and reliable.
- The protocol must enforce stale-price windows and require explicit oracle freshness checks.
- A fallback or emergency pause must exist if the oracle fails or becomes disconnected.

### 7.2 Permissioning

- PT, YT, and SY instruments must preserve underlying USDY eligibility.
- Transfer and redemption should be blocked for unauthorized participants.
- Protocol permissioning must not create a compliance bypass.

### 7.3 Emergency Controls

- `pause()` stops new issuance, trading, and redemption.
- `circuit_breaker()` halts risky actions when reference-price or liquidity invariants fail.
- `freeze()` may be used for proven compromise or regulatory demands.

### 7.4 Accounting and Rounding

- All value accounting must be anchored in USDC-equivalent reference values.
- The protocol must define deterministic rounding rules for minting, trading, and settlement.
- Settlement calculations should be audited for off-by-one and rounding drift.

## 8. Market Infrastructure

### 8.1 PT/SY Liquidity Pool

- Single pool type for PT versus SY positions.
- Liquidity providers earn swap fees.
- The pool supports deep PT liquidity and enables YT exposure through routing.

### 8.2 YT Market Interaction

- YT is priced indirectly via PT valuation and underlying SY reference value.
- The router uses path-based operations to mint or redeem YT against the underlying SY pool.
- This design reduces pool fragmentation.

### 8.3 Fees

- Protocol fees on YT yield claims.
- Swap fees on PT/SY trades.
- Fees are collected into a protocol treasury for operations and incentives.

## 9. Permissioned Asset Considerations

### 9.1 Eligibility Enforcement

- The protocol must verify transfer and holding eligibility for the underlying asset before allowing PT/YT issuance or transfers.
- Permission checks should be embedded in every contract that mints, burns, or transfers derivative assets.

### 9.2 Compliance Consistency

- Token metadata and contract design should make asset eligibility requirements explicit.
- If the underlying asset is permissioned, PT and YT derivatives should adopt the same restrictions so the protocol does not create a compliance bypass.
- If the asset is permissionless, the same core architecture still applies without permission gating.

## 10. Deployment and Integration

### 10.1 Stargate and Soroban Integration

- Use Soroban contract accounts for all protocol contracts.
- Integrate with Stellar Horizon / Soroban RPC for transaction submission.
- Use Stellar Asset Contract support for USDY and derived token classes.

### 10.2 Wallet and Interface Integration

- Support existing Stellar wallets for auth-entry signing.
- Provide a user flow for permissioned onboarding, deposit, PT purchase, YT purchase, and redemption.
- Display maturity, fixed yield, and implied yield clearly.

## 11. Assumptions and Constraints

- The initial launch is focused on USDY and its permissioned ecosystem.
- The protocol assumes an issuer-provided or institutional reference-value feed.
- PT and YT instruments are not intended for general-purpose unpermissioned trading in v1.
- The protocol is built on Stellar’s native low-fee, fast-settlement strengths.

## 13. Oracle Security and Trust Model

The protocol requires a reliable USDY/USDC reference-value oracle. Minimum requirements:

- Multi-source or issuer-signed feed with onchain submitters.
- `min_sources` and quorum aggregation; compute median or weighted mean.
- `max_stale_seconds` (recommended 600s) and explicit `timestamp` checks.
- Oracle failures or insufficient sources must trigger `RiskControl.pause()`.

Oracle payload fields (onchain): `value` (fixed decimals), `timestamp`, `source_id`, `signature` (when applicable).

## 14. Deterministic Settlement Formula

Settlement converts USDC-accounting amounts into underlying USDY tokens at maturity using the reference value $r$ (USDC per USDY). Let:

- $P$ = principal USDC value per PT (e.g., 100.00)
- $r$ = USDY reference value in USDC at maturity (e.g., 1.03)

Then the USDY amount delivered to PT holders per PT is:

$$\text{USDY}_{PT} = \left\lfloor \frac{P}{r} \right\rfloor_{q}$$

where $\lfloor\cdot\rfloor_{q}$ indicates floor with quantization to USDY token decimals (e.g., 7-9 decimals). Residual USDC value from rounding is credited to `settlement_reserve` and distributed per governance rules.

The YT holder USDY amount is computed from the total underlying position minus the PT allocation.

## 15. Maturity Definition and Identity

Each issuance must include explicit fields:

- `maturity_timestamp` (unix seconds)
- `reference_value_timestamp` (time at which oracle value is sampled)
- `issue_nonce` to ensure unique issuance ids

Child execution and redemption records must include the issuance id and `maturity_timestamp` to prevent cross-issuance replay.

## 16. Permissioning Interface

Provide a lightweight, auditable interface used by all contracts:

```rust
trait Permissioning {
   fn is_allowed(account: Address) -> bool;
   fn is_allowed_for_asset(account: Address, asset: Address) -> bool;
}
```

All mint, burn, transfer, and redemption entrypoints must call `Permissioning` before state changes.

## 17. Fees, Parameters, and Governance

- Define `fee_yield_bps` (basis points on yield claims) and `fee_swap_bps` (swap fee on PT/SY trades).
- Cap configurable governance parameters (e.g., `max_fee_bps = 200`) with timelock on parameter increases.
- Fee accrual accounting must be explicit and onchain to `protocol_treasury` address.

## 18. Rounding Rules and Residuals

- Use fixed-point arithmetic with a clear decimals constant per asset.
- For PT redemption: compute USDY amount as floor division, credit residual USDC value to `settlement_reserve`.
- For recombination (PT+YT -> SY): recombination must be exact when both tokens are present; require `recombination_nonce` to avoid race.
